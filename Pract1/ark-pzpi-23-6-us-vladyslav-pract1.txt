2 ЗАВДАННЯ
     Метою роботи є опанування основних правил оформлення та рефакторингу програмного коду на мові C відповідно до вимог GNU Coding Standards. У межах роботи передбачено ознайомлення з принципами написання чистого, читабельного та підтримуваного коду, дослідження рекомендацій щодо форматування, іменування, структурування модулів та документування функцій, а також демонстрація цих вимог на практичних прикладах. Додатковою метою є формування навичок аналізу якості коду та застосування інструментів автоматизованого форматування й статичного аналізу для підвищення надійності програмного забезпечення.

3 ОПИС ВИКОНАНОЇ РОБОТИ
     3.1 Вибір мови програмування та джерел
     Для виконання практичної роботи обрано мову програмування C, на ній пишуть операційні системи, компілятори, системні утиліти, мережеві служби та вбудоване програмне забезпечення.
     Під час дослідження використовувалися вимоги та рекомендації офіційної код-конвенції GNU Coding Standards, яка визначає порядок форматування, правила іменування, структурування файлів та підходи до написання зрозумілого й підтримуваного коду на мові C.
     3.2 Іменування змінних і функцій
     У GNU Coding Standards рекомендується використовувати зрозумілі та однозначні англомовні назви у форматі lowercase_with_underscores. Окремо визначено правило оформлення констант і макросів — їх записують у верхньому регістрі з підкресленнями, що забезпечує чітке відокремлення незмінних значень від інших елементів програми. Використання змістовних ідентифікаторів підвищує передбачуваність поведінки програми, зменшує кількість помилок та полегшує підтримку проєкту в майбутньому. Натомість короткі абстрактні назви ускладнюють аналіз та створюють ризик неправильного використання або модифікації функцій та змінних.
1 // Поганий приклад
2 int
3 f (int a, int b)
4 {
5   int t;
6   t = a * b * 2;   
7   return t;
8 }
9 
10 // Гарний приклад

12 const int BASE_VALUE = 1;

14 int
15 multiply_integers (int left, int right)

20   return product;

     3.3 Форматування коду, відступи та розміщення фігурних дужок
     GNU Coding Standards встановлює чіткі вимоги до форматування коду, що забезпечує його передбачувану структуру та полегшує подальший аналіз. Основним відступом є два пробіли, що дозволяє створювати компактну та рівномірну ієрархію вкладених блоків. Перед круглими дужками завжди має бути пробіл, включаючи виклики функцій, умовні оператори та цикли. Довжина рядка не повинна перевищувати 79 символів, що робить код зручним для перегляду в терміналі та в інструментах контролю версій. Характерною особливістю GNU-стилю є те, що тип функції розміщується окремим рядком над її назвою, а відкриваюча фігурна дужка ставиться в першій колонці нового рядка. Таке вирівнювання дозволяє інструментам автоматичного форматування та редакторам коректно визначати початок функції, а також значно полегшує навігацію у великих файлах. Дотримання цих правил мінімізує візуальний безлад, зменшує ризик помилок у вкладених конструкціях та позитивно впливає на швидкість налагодження й підтримку проєктів.
1 // Поганий приклад
2 int max(int a,int b){
3 if(a>b){return a;}else{return b;}}
4 
5 // Гарний приклад
6 int
7 max (int a, int b)
8 {
9   if (a > b)
10     {
11       return a;
12     }
13   else
14     {
15       return b;
16     }
17 }

     3.4 Коментарі до коду
     GNU Coding Standards наголошує на необхідності чітких та інформативних коментарів, особливо перед оголошеннями функцій. У мові C коментарі компенсують відсутність вбудованих описів поведінки функцій та типів, тому саме вони пояснюють контекст, правила використання вказівників, особливості обробки аргументів і повернення значень. Наявність коректних коментарів зменшує ризик неправильного використання функцій, покращує розуміння логіки та значно прискорює підтримку проєкту.
1 // Поганий приклад
2 int
3 string_length(char *s)
4 {
5   /* ... реалізація ... */ 
6   return 0;
7 }
8 
9 // Гарний приклад
10 /* string_length – compute the length of string S.
11    Returns the number of characters.  */
12 int
13 string_length (const char *s)
14 {
15   /* ... реалізація ... */ 
16   return 0;
17 }
     3.5 Організація модулів і заголовкових файлів
     GNU Coding Standards передбачає розділення інтерфейсу та реалізації: заголовкові файли (.h) містять оголошення функцій, тоді як файли реалізації (.c) містять їхній внутрішній код. Такий поділ спрощує структуру проєкту, робить модулі незалежними, дозволяє змінювати внутрішню логіку без впливу на інші частини програми та покращує читабельність і супровідність коду.
1 // Поганий приклад 
2 // Оголошення і реалізація змішані в одному файлі 
3 
4 int sum (int a, int b)
5 {
6   return a + b;
7 }
8 
9 // Гарний приклад
10 
11 // math.h — оголошення функції (інтерфейс)
12 int math_sum (int left, int right);
13 
14 // math.c — реалізація функції 
15 #include "math.h"
16 
17 int
18 math_sum (int left, int right)
19 {
20   return left + right;
21 }
     3.6 Інструменти для чистого коду 
     Для підтримання єдиного стилю оформлення, підвищення читабельності та виявлення дефектів коду використовуються такі утиліти командного рядка:
* indent - інструмент для автоматичного форматування C-коду відповідно до GNU-стилю. Приводить відступи, пробіли та розміщення дужок до узгодженого вигляду.
* clang-format - утиліта для автоматичного форматування великих проєктів за єдиними стильовими правилами. Забезпечує однакову структуру коду незалежно від кількості розробників.
* clang-tidy - інструмент статичного аналізу для виявлення логічних помилок, небезпечних конструкцій, неправильного використання пам’яті та порушень стильових вимог.
* cppcheck - утиліта статичного аналізу, яка виявляє дефекти, що можуть бути пропущені компілятором: неініціалізовані значення, потенційні витоки пам’яті, вихід за межі масивів та інші помилки низького рівня.

ВИСНОВКИ

	У ході виконання практичної роботи було розглянуто основні принципи оформлення та рефакторингу програмного коду на мові C відповідно до вимог GNU Coding Standards. Опрацьовані правила форматування, іменування, коментування та організації модулів дозволили сформувати цілісне уявлення про те, яким має бути структурований, зрозумілий і підтримуваний програмний код. 
     Отримано практичні навички аналізу фрагментів коду та визначення недоліків, що впливають на читабельність і якість програми. Виконано рефакторинг прикладів, що дало змогу закріпити підхід до покращення структури програмного коду та підвищення його зрозумілості. 
     У процесі роботи було опрацьовано методи автоматизованого форматування та статичного аналізу, що продемонструвало переваги використання відповідних інструментів для забезпечення стабільного стилю, зменшення кількості помилок та покращення загальної якості програмного забезпечення. Застосування таких інструментів підтвердило важливість єдиних стильових стандартів та системного підходу до контролю якості коду.
